# Pokemon Champions 構築提案プロトコル (HARD-GATE)

> **目的**: 構築提案で数字根拠を欠いた誤りを防ぐ。
> 動画偏重の感覚提案ではなく、DB の採用率データを必須参照する手順。

---

## 全体フロー

```
[Step 1] 環境把握
  └─ champs.pokedb / pokechamdb / yakkun から TOP30 と全体動向取得

[Step 2] 個別ポケのデータ収集
  └─ 提案する 6 体それぞれについて:
      - champs.pokedb 個別ページ (持ち物/特性/性格/技/努力値の採用率)
      - pokechamdb クロスチェック
      - yakkun 育成論ページで補強

[Step 3] 実装確認
  └─ implementation_status.md で持ち物・技がナーフ/未実装じゃないか確認

[Step 4] 動画 + X で「なぜ」を補完
  └─ クロコ/今日ポケch./ポケソル/KYOUPOKE/note 記事 24-72h 以内の動向

[Step 5] 構築提案テンプレを埋める
  └─ data_extraction_guide.md の YAML テンプレで 6 体分

[Step 6] 私の選択 vs 採用率 のチェック
  └─ 各選択が採用率 30%+ なら主流、未満ならマイナー扱いを明示

[Step 7] ユーザーに提案 (鮮度・根拠ソース必須明記)
```

## 採用率データの読み方 (対策地図 / 使い方学習 / 自パーティ補完)

> **原則**: 採用率は「主流 / マイナー」のラベル付けではなく、
> **対策地図 (環境理解) と 使い方学習 (個別研究) の二層** で読む。
> 用語の SSOT: `SKILL.md` 第 14 節。

### Step A: 対策地図として読む (環境把握)

全体使用率 TOP30・各ポケの持ち物/技 TOP は **「環境で対策される対象」** として読む。

| 採用率 | 環境的意味 | 自パ提案への活用 |
|-------|-----------|----------------|
| 50%+  | 全プレイヤーが対策必須 | 自パに該当ポケへの回答役 1 体は必ず入れる |
| 30-50%| 想定対面に頻出 | 想定対面の上位 6 個に含めて検討 |
| 15-30%| 当たれば不利、対面落としは許容圏 | パ全体で軽い回答があれば OK |
| 5-15% | レア対面、対策枠割く必要なし | スルー |

### Step B: 使い方学習として読む (個別研究)

そのポケの技/持ち物 TOP は **「そのポケの定石立ち回り」** を学ぶ目的で見る。
**実採用の判断には使わない**。

### Step C: 自パーティ補完で実採用を決定

実際に技 4 つを決めるときは、Step B の定石を参考にしつつ、
**自パ 6 体で既に埋まっている役割を除いて** 残り枠を選ぶ。

```
例: ガブ採用、自パに既にステロ持ちカバが居る
→ 定石ステロ 51.7% は不要、岩石封じ or どくづき の枠へ
→ 自パ S 操作なら岩石封じ、毒打点欠なら どくづき
```

### 旧基準 (中堅帯向け、参考保持)

<details>
<summary>採用率単一軸ラベル (旧)</summary>

| 採用率 | ラベル | 提案時の表現 |
|-------|--------|-------------|
| 50%+ | 圧倒的主流 | 「ほぼ全採用」 |
| 30-50% | 主流 | 「主流の選択肢の一つ」 |
| 15-30% | 準主流 | 「選択肢として有力」 |
| 5-15% | マイナー | 「敢えてのマイナー選択、理由を明示」 |
| < 5% | レア | 「ロマン枠」 |

※ハイパー帯以上では Step A-C の二層解釈を優先。
</details>

## 動画 vs DB の使い分け

| 情報種別 | 一次ソース (信頼度高) | 補完ソース |
|---------|----------------------|----------|
| 使用率 TOP | DB 系 | 動画ティアの裏取り |
| 持ち物採用率 | DB 系 | (動画は感覚値) |
| 技採用率 | DB 系 | 動画解説で「なぜ」 |
| 性格採用率 | DB 系 | (動画ほぼ言及なし) |
| 環境メタ感覚 | 動画 (クロコ/ポケソル) | DB は数字のみ |
| 新興構築論 | note記事/動画 | DB は反映遅れ |
| 個別ポケ評価 | 動画 tier発言 | DB は使用率のみ |
| 実装/未実装 | yakkun (公式) / アルテマ未実装一覧 | 動画はバラバラ |
| ナーフ/バフ | yakkun changes.htm | X @nomaneko_world / @poketettei |

**鉄則**: 数字 (%) は DB、評価は動画、両者整合性をチェック。

## アンチパターン (やってはいけない)

### ❌ 動画だけで構築組む
```
悪い例: 「クロコ氏が S 級と言ったから採用」
良い例: 「クロコ氏が S 級と評価 + DB で使用率 1 位 + 採用率 99% さめはだ」
```

### ❌ DB データの last_updated を確認しない
```
悪い例: 7 日前の DB データで「現環境は…」と語る
良い例: 「DB 4/28 23:44 更新、X 直近 24h で大変動なし → 信頼度高」
```

### ❌ 採用率の低い型を主流扱い
```
悪い例: 「メガカイリューは物理いじっぱりがテンプレ」
       (実際は ひかえめ 70.8% / いじっぱり 15.5%)
良い例: 「メガカイリュー主流は ひかえめ CSぶっぱ (70%)。
        いじっぱり物理は 15% のサブ型」
```

### ❌ 実装確認せず未実装アイテムを提案
```
悪い例: 「カバルドンに ゴツゴツメット を持たせる」
       (ゴツメ未実装)
良い例: 「カバルドンに オボンのみ (採用率 67%) を持たせる」
```

### ❌ ナーフ前の感覚で技を選ぶ
```
悪い例: 「アシレーヌのムンフォ起点で C 下げ狙う」
       (ムンフォの C 下げ確率 30%→10% にナーフ)
良い例: 「アシレ ムンフォの C 下げは 10% で偶発、
        ハイドロ主軸推奨」
```

### ❌ 高採用率ポケ・技を「強い」と読む (採用率 = 強さ誤読)
```
悪い例: 「ガブ使用率1位 → 最強 → 主軸据える」
良い例: 「ガブ使用率1位 = 全員が対策積んでる → 自パに『ガブ越え手段』必須。
         主軸ではなく『対策される前提でどう刺すか』で採用検討」
```

### ❌ 2 DB 一致を「信頼度高」の根拠にする
```
悪い例: 「champs.pokedb と pokechamdb で技採用率が一致 → データ信頼できる」
良い例: 「2 DB は同じ Champions 公式統計系 → 一致は当然。
         一致は『事実確認の場』(パース誤差検出) であって信頼度根拠ではない。
         真の信頼度は note/動画の強プレイヤー構築事例で別途検証」
```

### ❌ 技採用率 TOP6 を実採用判断に使う
```
悪い例: 「ガブの技 TOP4 をそのままコピー」
良い例: 「ガブの技 TOP6 は『使い方学習』(立ち回り定石把握) のために見る。
         実採用の 4 技は自パ補完原則で決める (既に埋まってる役割は除外)」
```

## 実例: ガブリアスの正しい提案手順

```
1. データ取得
   - champs.pokedb /pokemon/show/0445-00?rule=0
   - pokechamdb /en/pokemon/garchomp
   → 取得結果:
     - 持ち物: タスキ40.5% / スカーフ30.6%
     - 性格: ようき69.9%
     - 技: じしん98% / げきりん61% / ステロ51.7% / 岩石封じ43%
     - EV: H4/A252/S252 = 55.3%

2. 実装確認 (yakkun)
   - タスキ ✓ / げきりん ✓ / ステロ ✓ / 岩石封じ ✓

3. 動画補強 (クロコ 4/26 + ポケソル 4/28)
   - クロコ: ガブは S 級唯一
   - ポケソル: お盆ガブが激増 (ステロ + 岩石封じ起点)

4. 私の構築 (剣舞型) vs データ
   - 剣舞 → 採用率 5% 未満 (技 TOP6 圏外)
   - 「マイナー型」確定

5. 修正提案
   - 主流型: タスキ × ようき AS × じしん/げきりん/ステロ/岩石封じ
   - 「ステロ起点作り」役 = ガブの環境的な立ち位置
   - 剣舞型は所持ポケに合わせる場合のサブ案として併記
```

## ハイパー級ユーザー向けの読み方プロセス

### A. 環境把握フェーズ — 対策地図として読む

```
[ ] TOP30 を「環境で対策が積まれている対象」のリストとして書き出す
[ ] TOP10 各々について「自パにこのポケへの回答役はあるか」を 1 列追加
[ ] 回答役欠落ポケを「自パの穴」として 3 つ以内に絞り込む
[ ] 穴を埋めるための候補ポケを 5-10 体出す (ここで採用率は見ない)
```

### B. 個別研究フェーズ — 使い方学習として読む

```
[ ] 候補ポケ 5-10 体それぞれの DB を「立ち回り定石」として読む
    (技 TOP6 = どう動かすのが定石か / 持ち物 TOP3 = 定石対応)
[ ] 「実採用の根拠」にはまだ使わない
```

### C. 自パ補完フェーズ — 実採用判断

```
[ ] 自パ 5 体で既に埋まっている役割を列挙
[ ] 候補ポケの定石技から「埋まっている役割」を除外
[ ] 残った枠で「自パで足りない役割」を埋める技を選ぶ
[ ] 結果が定石 (採用率 TOP) と一致しなくても OK
    (むしろ「読まれない型」=対策外しになる場合あり)
```

### D. 既存配慮 (会話表現の平易さ) — 保持

```
[ ] 専門用語が出たら初登場時に括弧書きで説明
    例: 起点作り(=積み技を使うための盤面準備)
[ ] 各ポケに「役割キーワード」を 2 つだけ付与
    例: ガブ = 「起点作り役」「対面壊し役」
[ ] 選出パターンを 3-5 通りに圧縮
[ ] 「迷ったらこの 1 つ」フォールバック選出を提示
[ ] タイプ相性は「弱点 4 つ」だけ覚えれば OK の形に圧縮
```
